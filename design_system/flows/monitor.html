TASK FLOW

Monitoring Tasks / Experimental Events / Data

NOTE: This is a work in progress and is expected to frequently change.

Intent of task flow:

Task monitors help users track tasks and make decisions based on the status of those tasks. This pattern can be used while running an experiment, in which case, one such decision could be to stop the experiment entirely (in case there are problems with the instrument). In instances where the data monitor is used asynchronously, it can be used to determine if the task should be run again, in cases of failures.

It is different from other monitoring systems in that it is meant to track ongoing tasks, not the status of a system (which is usually represented by a data dashboard).

Task flow example template

Do's:

  • Real-time experiments
    • The display should refresh to show new data as the data becomes available
    • If data or events need to be reviewed, highlight the new data/events to draw attention to the ones needing review.
    • Enable an easy way for users to add notes alongside monitored events
    • If the monitor is to be used alongside other software, consider a smaller aspect ratio app (similar to a chat program) for monitoring events, using the desktop ratio version for more detailed work.
  • Asynchronous monitors
    • Usage of states can be useful to help determine next steps. For example, in a job monitor, some indication if the job failed is helpful to determine if the job should be rerun. In cases of experimental observations, the state can determine if more observations are needed.
    • Post task monitors should at the first step be a simple list but can allow access to much more data dense views than in real-time monitors, since time is not as constrained. Rich data access can help users determine next steps.
